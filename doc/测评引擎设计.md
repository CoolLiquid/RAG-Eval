# 测评引擎设计 — 知识库测评平台

**文档版本**：v0.1  
**更新日期**：2026-07-02  
**状态**：草案

---

## 1. 概述

测评引擎负责在 MCP 知识库挂载完成后，按题库批量执行检索（及可选生成）、评分、归因，并输出结构化报告数据。

### 1.1 输入 / 输出

```
输入：
  - MCP 知识库实例（endpoint, tool, auth）
  - 测评题库（questions + expected_answers）
  - 测评配置（mode, top_k, threshold, scoring_method）

输出：
  - 任务级报告（综合得分、通过率、指标分布）
  - 题目级结果（得分、片段、MCP 证据、失败类型）
```

## 2. 测评模式

### 2.1 检索测评（retrieval_only）— MVP 默认

**目标**：评估 MCP 检索 Tool 能否召回正确知识片段。

**流程**：

```
question
  → MCP kb_search(query, top_k)
  → chunks[]
  → 评分：检查召回 + 片段相关性
  → result
```

**通过判定（默认）**：

1. 若题目有 `source_ref`：Top-K 结果的 `source` 中包含 `source_ref` → 召回成功
2. 最高相关片段与 `expected_answer` 语义相似度 ≥ 阈值（默认 0.75）
3. 两项均满足 → 通过

### 2.2 RAG 全链路测评（rag_full）— V1.1

**目标**：评估检索 + 生成整体问答质量。

**流程**：

```
question
  → MCP kb_search(query, top_k)
  → chunks[]
  → LLM generate(question, chunks) → answer
  → LLM judge(question, expected, answer, chunks) → score
  → result
```

**通过判定**：LLM 裁判 `score ≥ threshold` 且 `failure_type` 非 hallucination / wrong_answer。

## 3. 任务状态机

```
                    ┌──────────┐
                    │ pending  │
                    └────┬─────┘
                         │ start
                         ▼
                    ┌──────────┐
              ┌─────│ running  │─────┐
              │     └────┬─────┘     │
         cancel│          │          │ all done
              │          │ error     │
              ▼          ▼           ▼
        ┌──────────┐ ┌────────┐ ┌───────────┐
        │ cancelled│ │ failed │ │ completed │
        └──────────┘ └────────┘ └───────────┘
```

| 状态 | 说明 | 可转移至 |
|------|------|----------|
| pending | 已创建，等待 Worker | running |
| running | 逐题执行中 | completed / failed / cancelled |
| completed | 全部题目处理完毕 | — |
| failed | 系统性错误（如 MCP 全局不可达） | — |
| cancelled | 用户取消 | — |

**进度计算**：

```
progress = floor(completed_questions / total_questions * 100)
```

## 5. 评分策略

### 5.1 检索测评评分（retrieval_only）

```python
def score_retrieval(question, expected, chunks, source_ref, threshold):
    if not chunks:
        return Score(0, False, "no_recall", "未检索到任何片段")

    # 召回检查
    if source_ref:
        recalled = any(source_ref in c.source for c in chunks)
        if not recalled:
            return Score(0, False, "no_recall", f"未召回期望来源: {source_ref}")

    # 语义相似度（取 top-1 或最高 score 片段）
    best_chunk = max(chunks, key=lambda c: c.score or 0)
    similarity = semantic_similarity(best_chunk.content, expected)

    if similarity >= threshold:
        return Score(similarity * 100, True, None, "召回且内容相关")
    else:
        return Score(similarity * 100, False, "wrong_answer", "召回片段与期望答案不匹配")
```

### 5.2 LLM 裁判评分

**Prompt 模板**：

```
你是一个 RAG 系统测评裁判。请根据以下信息评判回答质量。

## 问题
{question}

## 期望答案
{expected_answer}

## 检索到的知识片段
{chunks_formatted}

## 实际回答（如有）
{generated_answer}

## 评判规则
- 如果检索片段中不包含回答问题的关键信息，failure_type = no_recall
- 如果回答与期望答案语义一致，passed = true
- 如果回答包含片段未支持的信息，failure_type = hallucination
- 如果部分正确但不完整，failure_type = incomplete
- 如果片段相关但回答错误，failure_type = wrong_answer

请输出 JSON：
{
  "score": 0-100,
  "passed": true/false,
  "failure_type": null | "no_recall" | "wrong_answer" | "hallucination" | "incomplete",
  "reason": "简要说明"
}
```

**参数**：`temperature=0`，保证评分稳定性。

### 5.3 语义相似度评分

```
score = cosine_sim(embed(question + expected), embed(chunk_content)) * 100
passed = score >= threshold
```

适用于不想依赖 LLM 裁判的场景，成本更低但精度略差。

## 6. 失败归因

| failure_type | 触发条件 | 建议修复方向 |
|--------------|----------|--------------|
| `mcp_error` | MCP 调用异常、超时、鉴权失败 | 检查 MCP 服务、网络、密钥 |
| `no_recall` | 无片段或 source_ref 未命中 | 优化切分、Embedding、索引 |
| `wrong_answer` | 有召回但内容不匹配 | 检查检索排序、Top-K 参数 |
| `hallucination` | 生成内容超出片段支持范围 | 优化 Prompt、降低温度 |
| `incomplete` | 部分正确 | 增加 Top-K 或优化切分粒度 |
| `mapping_error` | 响应解析失败 | 检查 Tool 映射配置 |

## 7. 报告指标

### 7.1 任务级指标

| 指标 | 公式 | 说明 |
|------|------|------|
| 综合得分 | `avg(question.score)` | 0–100 |
| 通过率 | `passed_count / total * 100%` | — |
| 召回率 | `非 no_recall 题数 / total * 100%` | 检索模式核心指标 |
| 幻觉率 | `hallucination 题数 / total * 100%` | RAG 模式 |
| MCP 成功率 | `非 mcp_error 题数 / total * 100%` | 服务稳定性 |
| 平均 MCP 时延 | `avg(latency_ms)` | 性能 |
| P95 MCP 时延 | `percentile(latency_ms, 95)` | 性能 |

### 7.2 分组指标

- 按 `category` 分组：类目得分、通过率
- 按 `difficulty` 分组：难度分布
- 按 `failure_type` 分组：失败原因占比

### 7.3 版本对比（V1.1）

同一题库，对不同 MCP 知识库实例或不同配置（top_k、Tool 版本）的两次测评：

| 对比项 | 说明 |
|--------|------|
| 综合得分 delta | A vs B 分数差 |
| 通过率 delta | — |
| 逐题 diff | 同一题在 A 通过、B 失败（或反之） |
| MCP 时延对比 | — |

## 8. 并发与容错

### 8.1 并发模型

```
EvaluationTask
  → Worker Pool (默认 concurrency=3)
  → 每 Worker 独立 MCP 会话
  → 逐题串行（单 Worker 内），多 Worker 并行
```

**注意**：并发度应可配置，避免压垮 MCP 服务。

### 8.2 超时与重试

| 配置 | 默认值 | 说明 |
|------|--------|------|
| 单题 MCP 超时 | 30s | 超时 → mcp_error |
| MCP 重试次数 | 1 | 仅网络超时重试 |
| 任务级超时 | 30min | 整体任务标记 failed |

### 8.3 部分失败

- 单题失败不阻断任务
- 报告标注成功/失败题数
- MCP 全局不可达（连续 N 题 mcp_error）→ 任务 failed

## 9. 数据存储

### 9.1 EvaluationResult 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| task_id | UUID | 关联任务 |
| question_id | UUID | 关联题目 |
| mcp_tool | string | 调用的 Tool 名 |
| mcp_request | JSONB | 请求参数 |
| mcp_response_raw | JSONB | 原始响应 |
| chunks | JSONB | 归一化片段 |
| generated_answer | text | RAG 模式生成答案 |
| score | float | 0–100 |
| passed | boolean | 是否通过 |
| failure_type | string | 失败类型 |
| failure_reason | text | 失败说明 |
| latency_ms | int | MCP 调用耗时 |
| created_at | timestamp | — |

### 9.2 原始响应存储策略

- 全量存储于 `mcp_response_raw`（MVP）
- 响应 > 100KB 时截断并标记 `truncated: true`
- 保留 90 天（可配置）
